- 0 → the interaction MUST explore JOIN combinations involving the target table.
- 1 → the interaction MUST explore COLUMN combinations of the target table (no new joins).

You MUST strictly follow the binary interaction list in order. Do NOT change its length, order, or meaning.

{interaction_binary_list}

{target_table}

## Dialogue Generation Rules

- Generate exactly {total_expected_interactions} interactions.
- Interaction i MUST follow the directive at interaction_binary_list[i].
- All interactions are user utterances ("speaker": "User").
- Each interaction must be a natural continuation of the previous one (multi-turn dialogue).
- Follow-up questions MUST depend on previous answers.
- Do NOT mix join-based reasoning and column-based reasoning in the same interaction.
- Use ONLY the provided join or column combinations; do not invent schema elements.
- Do NOT write a dialogue interaction referencing "target_table", use it's real name instead when the case.
- Ensure semantic diversity across interactions.
- Avoid repeating query intent patterns.

### Binary Rules Recap
- If interaction_binary_list[i] == 0:
  - The question must require JOIN reasoning.
  - Use join conditions from the Join Combination Context.
- If interaction_binary_list[i] == 1:
  - The question must focus on filtering, aggregation, grouping, or ordering using columns.
  - Do NOT introduce new joins.
  - golden_sql must be context-correct: include any JOINs required by entities/constraints introduced in previous interaction, 
even if the current interaction was generated under a column-based directive.

## Field Constraints

- "utterance":
  - Natural, conversational user question.
- "intention":
  - A refined natural language rewrite of the utterance.
  - Must NOT reference database tables, columns, or schema.
  - For follow-ups, include context implicitly (e.g., “that team”, “this country”).
- "tables_from_schema_linking":
  - Exact table names used in the SQL.
- "golden_sql":
  - Valid SQL executable on the given schema.
  - Use ONLY tables and columns provided.
  - Use UPPERCASE for table and column names (e.g., MONDIAL_CITY.NAME).
  - Sample values are allowed ONLY as filters.
